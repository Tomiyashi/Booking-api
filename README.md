### Booking Service API (Avito Internship)
Сервис бронирования переговорок для сотрудников. Позволяет администраторам управлять ресурсами и расписанием, а пользователям — бронировать доступные временные интервалы.

## Технологический стек
Language: Go 1.22

Database: PostgreSQL

Containerization: Docker, Docker Compose

API Spec: OpenAPI (api.yaml)

## Как запустить
Сборка и запуск:

Bash
docker-compose up --build
Проверка работоспособности:
Сервис будет доступен по адресу http://localhost:8080. Проверить статус можно эндпоинтом:

Bash
curl http://localhost:8080/_info
Основные архитектурные решения
## Генерация слотов (Бизнес-логика)
Выбран подход предварительной генерации (Pre-generation) при создании расписания администратором.

Почему: ТЗ требует выполнения SLI 200мс на самом нагруженном эндпоинте (получение списка слотов). Предварительная генерация позволяет базе данных отдавать готовые строки по индексу, не тратя ресурсы CPU на вычисления интервалов «на лету» при каждом запросе пользователя.

UUID: Каждый слот сохраняется в БД с фиксированным UUID, что гарантирует стабильность ссылок при бронировании.

## Производительность и SLI
Для достижения высокой скорости ответа (цель — <200 мс) реализовано:

Индексация: Составной индекс idx_slots_time (room_id, start_time) позволяет мгновенно фильтровать свободные интервалы на конкретную дату.

SQL Query: Поиск свободных слотов выполняется через LEFT JOIN с таблицей бронирований, исключая занятые (status = 'active').

## Гарантии бронирования и Идемпотентность
Защита от Double-booking: На уровне БД реализован частичный уникальный индекс:
UNIQUE(slot_id) WHERE (status = 'active').
Это позволяет иметь историю отмененных броней, но гарантирует, что в один момент времени на один слот может быть только одна активная бронь.

Идемпотентность отмены: Метод отмены брони переводит статус в cancelled. Повторные вызовы не создают ошибок и возвращают текущее состояние ресурса (200 OK).

## Безопасность и Роли
JWT: Система ожидает токен с полями user_id и role.

Доступ: * Эндпоинты /admin/** доступны только для role: admin.

Бронирование доступно только для role: user. Администраторы не могут занимать переговорки.

Структура БД
Rooms: Справочник переговорок.

Slots: Сгенерированные 30-минутные интервалы.

Bookings: Связь пользователя и слота со статусом (active/cancelled).
